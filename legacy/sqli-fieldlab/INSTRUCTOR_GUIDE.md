# FieldLab 教师指南

> 说明：本指南面向授课教师。学生版界面只提供渐进式提示，不直接给完整解法。

# 一、先讲分类框架，再进关卡

本靶场采用 **1 个主轴 + 4 个副轴**：

## 主轴：按注入位置分类
1. 值上下文注入
2. 结构上下文注入
3. 语句级注入
4. 动态 SQL 注入

## 副轴 1：按生效时机分类
- 一次注入
- 二次注入

## 副轴 2：按观察方式分类
- 直接回显型
- 报错型
- 布尔盲注
- 时间盲注
- 带外潜力型

## 副轴 3：按最终影响分类
- 认证绕过
- 数据读取
- 数据枚举
- 数据篡改
- 语句扩展
- 错误修复对照

## 副轴 4：按数据库方言分类
- MySQL / MariaDB
- PostgreSQL
- SQL Server
- Oracle
- SQLite

# 二、先按主轴组织授课

## 1. 值上下文注入
对应关卡：
- L01 登录网关绕过
- L02 商品检索 UNION 泄露
- L03 工单详情报错注入
- L04 门禁查询布尔盲注
- L05 物流接口时间盲注
- L10 JSON API 参数注入
- L11 伪修复：黑名单过滤

教学目标：
- 让学生先理解“输入本应只是值，却跑进了 SQL 逻辑里”。
- 这是所有 SQLi 的起点。

## 2. 结构上下文注入
对应关卡：
- L08 排行榜排序 / LIMIT 注入

教学目标：
- 强调“不是所有输入都是值位”。
- 讲清楚为什么 `ORDER BY`、`ASC/DESC`、`LIMIT` 不能只靠参数化。

## 3. 语句级注入
对应关卡：
- L06 审计报表堆叠查询
- L09 成绩调整 UPDATE 注入

教学目标：
- 把学生从“读数据”带到“改数据 / 扩大执行范围”。
- 让学生理解 SQLi 对业务完整性的破坏。

## 4. 动态 SQL 注入
对应关卡：
- L07 二次注入：保存筛选器
- L12 ORM 误用

教学目标：
- 讲清动态 SQL 的风险。
- 让学生知道 ORM、数据库里已有数据都不是天然可信边界。

# 四、逐关简表

## L01 登录网关绕过
- 主轴：值上下文注入
- 时机：一次注入
- 观察：直接回显型
- 影响：认证绕过
- 讲法：用最简单的认证语句让学生理解“输入改写逻辑”。

## L02 商品检索 UNION 泄露
- 主轴：值上下文注入
- 时机：一次注入
- 观察：直接回显型
- 影响：数据读取
- 讲法：把 UNION 明确为“观察方式 / 取数方式”，不要混成“注入位置”。

## L03 工单详情报错注入
- 主轴：值上下文注入
- 时机：一次注入
- 观察：报错型
- 影响：数据枚举
- 讲法：说明“详细错误回显”为什么是高价值情报通道。

## L04 门禁查询布尔盲注
- 主轴：值上下文注入
- 时机：一次注入
- 观察：布尔盲注
- 影响：数据枚举
- 讲法：真假响应也是数据通道。

## L05 物流接口时间盲注
- 主轴：值上下文注入
- 时机：一次注入
- 观察：时间盲注
- 影响：数据枚举
- 讲法：把“无回显”升级成“有侧信道”。

## L06 审计报表堆叠查询
- 主轴：语句级注入
- 时机：一次注入
- 观察：直接回显型
- 影响：语句扩展
- 讲法：强调驱动和 multi-statements 配置的关键性。

## L07 二次注入：保存筛选器
- 主轴：动态 SQL 注入
- 时机：二次注入
- 观察：直接回显型
- 影响：数据读取
- 讲法：第一次存储，第二次触发。

## L08 排行榜排序 / LIMIT 注入
- 主轴：结构上下文注入
- 时机：一次注入
- 观察：直接回显型
- 影响：数据枚举
- 讲法：白名单是重点，不是参数化。

## L09 成绩调整 UPDATE 注入
- 主轴：语句级注入
- 时机：一次注入
- 观察：直接回显型
- 影响：数据篡改
- 讲法：SQLi 不只是读，还会改业务事实。

## L10 JSON API 参数注入
- 主轴：值上下文注入
- 时机：一次注入
- 观察：直接回显型
- 影响：数据读取
- 讲法：JSON 只是载体，不是安全边界。

## L11 伪修复：黑名单过滤
- 主轴：值上下文注入
- 时机：一次注入
- 观察：直接回显型
- 影响：错误修复对照
- 讲法：用来回收全课，强调黑名单不解决边界问题。

## L12 ORM 误用
- 主轴：动态 SQL 注入
- 时机：一次注入
- 观察：直接回显型
- 影响：数据读取
- 讲法：讲“用了 ORM”不等于“没有 SQLi”。

# 五、数据库差异怎么穿插讲

本靶场运行在 MariaDB，但建议授课时这样穿插：

- L05 时讲时间盲注函数差异：MySQL `SLEEP`、PostgreSQL `pg_sleep`、SQL Server `WAITFOR DELAY`、Oracle 常见 `DBMS_LOCK.SLEEP`
- L06 时讲堆叠查询不只看数据库，还看驱动/连接器配置
- L08 时讲结构位问题在各家数据库都存在，因为本质是“标识符 / 关键字不可参数化”
- L12 时讲 ORM 误用是跨数据库的，不依赖特定方言

# 六、统一修复原则

1. 值位置：参数化
2. 结构位置：白名单
3. 动态 SQL：避免拼接，或对可控片段做严格映射
4. 错误处理：前台隐藏细节，后台保留日志
5. 权限控制：最小权限
6. 驱动特性：禁用不必要的 multi-statements
7. 生命周期：数据库里的数据也不天然可信
